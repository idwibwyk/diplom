# Отчёт по созданию веб-приложения MARS GROOM  
## Модули «Курсы» и «Услуги»

# ЧАСТЬ 1. МОДУЛЬ «КУРСЫ»

## 1. Разработка фронтенда модуля курсов

Фронтенд модуля курсов веб-приложения MARS GROOM реализован на React с использованием TypeScript, библиотеки Motion для анимаций, React Router для маршрутизации и Tailwind CSS для стилей. Модуль курсов охватывает витрину образовательных программ груминга, детальные страницы курсов, расписание стартов и запись на курсы.

**Страницы и компоненты.** Главная точка входа — страница «Курсы» (CoursesMainPage): лендинг с hero-блоком на зелёной фирменной теме (#40AB40, #89E689), фоновым изображением, призывом к действию «Список всех курсов» и «Запись на курс». Ниже размещены блок «Путь грумера» (GroomerPath), секция с отзывами по курсам (карусель), блок теста и форма обратной связи (ContactForm). Страница списка курсов (CoursesPage / список) выводит карточки курсов; данные могут подгружаться с бэкенда через универсальный хук useEntity для сущности `courses`. Страница детального просмотра курса (CourseDetailPage) показывает полное описание одного курса: название, уровень (beginner/advanced), формат (online, offline, hybrid), длительность, цену, описание, изображение, программу (модули и контент), преподавателя (через связь course_instructors и masters), отзывы и кнопки «В избранное» и «Записаться». Для загрузки данных используются несколько экземпляров useEntity: курсы (courses) по id, расписание (course_schedules) по course_id, отзывы (reviews) по course_id и type, преподаватели (course_instructors), мастера (masters), модули (course_modules). Это обеспечивает единообразную работу с API и актуальные данные из БД. Страница расписания (CourseSchedulePage) отображает ближайшие даты стартов курсов с фильтром по формату (все / онлайн / очно / гибрид); данные о курсах и слотах могут браться из mockData или в перспективе из API course_schedules с привязкой к курсам. Страница записи на курс (CourseBookingPage) содержит форму выбора курса, даты и времени, контактных данных и отправляет заявку (course_bookings). Дополнительно в модуле задействованы компоненты CoursesGallery, CoursesHeader, CoursesFooter, FavoriteButton (для избранных курсов favorite_courses), общие ContactForm и GroomerPath.

**Технологии и приёмы.** На фронтенде используются хуки useState, useEffect, useMemo для состояния и производных данных; типизация TypeScript для сущностей (CourseRow, ScheduleRow, ReviewRow, MasterRow и т.д.); единый API-клиент (api.get, api.post) с базовым URL и JWT в заголовке Authorization; обработка состояний загрузки (loadingList, loadingItem) и ошибок (loadingListError, loadingItemError) с отображением индикаторов и сообщений пользователю. Стилизация выдержана в зелёной цветовой схеме курсов, адаптивная вёрстка и тёмная тема (dark:) учтены. Таким образом, фронтенд модуля курсов обеспечивает просмотр каталога, детальную информацию по курсу, расписание и запись, с опорой на единый хук useEntity и общие компоненты приложения.

**Детали реализации.** Маршруты модуля курсов регистрируются в корневом роутере приложения: /courses (главная), /courses/list (список), /courses/schedule (расписание), /courses/:id (детальная страница), /book/course (запись). Переход между страницами выполняется через компонент Link или navigate из React Router. Данные с сервера приходят в формате { success: true, data: … } или { success: false, error: "…" }; хук useEntity различает эти случаи и записывает данные в list/item или выставляет loadingListError/loadingItemError. На детальной странице курса блок программы строится по данным course_modules и course_content: модули отображаются в виде аккордеона или списка с длительностью и типом контента (видео, текст). Кнопка «В избранное» использует сущность favorite_courses и при наличии авторизации отправляет POST или DELETE на соответствующий эндпоинт. Анимации появления блоков (motion.div с initial/animate/transition) улучшают восприятие страницы без перегрузки контентом. Итоговая структура фронтенда модуля курсов позволяет пользователю познакомиться с программой обучения, выбрать курс и дату старта и оставить заявку на запись, при этом вся бизнес-логика загрузки и обновления данных сосредоточена в универсальном хуке и общем API-клиенте.

## 2. Разработка универсального контроллера и создание БД (модуль курсов)

Универсальный CRUD-контроллер и схема БД используются для всех сущностей приложения, в том числе для модуля курсов. Роутеры курсов и связанных сущностей создаются через фабрику createEntityRouter и единый universalController, что даёт согласованное API и разграничение прав по ролям.

**Таблицы БД, относящиеся к курсам.** В миграции создаются таблицы: `courses` (id, name, level, format, duration, price, description, image, created_at, updated_at, created_by); `course_schedules` (id, course_id, start_date, start_time, spots, timestamps, created_by); `course_instructors` (id, course_id, master_id, timestamps, created_by) с уникальной парой (course_id, master_id); `course_bookings` (id, user_id, course_id, course_schedule_id, master_id, status, notes, timestamps, created_by); а также таблицы модулей и контента курсов: `course_modules` (id, course_id, title, description, sort_order), `course_content` (id, module_id, title, type, content, file_path, duration_minutes), `user_course_progress`, `course_quiz_questions`, `course_quiz_options`, `course_quiz_answers`. Для избранного — `favorite_courses` (user_id, course_id). Связи заданы через внешние ключи (references) на users, masters, courses, course_schedules с соответствующими onDelete (CASCADE, SET NULL, RESTRICT). Общие для всех таблиц поля: created_at, updated_at, created_by (nullable), что поддерживается универсальным контроллером при создании и обновлении записей.

**Настройка прав для курсов в контроллере.** Роутер courses создаётся вызовом createEntityRouter('courses', { guest: { read: true, create: false, update: false, delete: false }, client: { read: true, create: false, update: false, delete: false }, groomer: { read: true, create: true, update: true, delete: false } }). Таким образом, гости и клиенты могут только читать список и детали курсов; создание и изменение доступны грумерам; удаление отключено для всех (курсы не удаляют через API). Универсальный контроллер по таблице courses реализует методы getAll (список с пагинацией и фильтрами по query), getById (одна запись по id), create, update, delete; перед каждой операцией выполняется проверка прав checkPermission по роли пользователя (req.user задаётся middleware optionalAuth). Для course_schedules, course_instructors, course_bookings, course_modules и других сущностей курсов используются отдельные роутеры с той же фабрикой и своими настройками прав (например, course_bookings с ownerField: 'user_id' для чтения/обновления своих записей клиентом). Создание БД выполняется миграциями Knex (migrate:latest); тестовые данные для курсов, расписания, преподавателей и записей заносятся сидами (seeds), что позволяет сразу проверять API и фронтенд модуля курсов.

## 3. Тестирование в Postman (модуль курсов)

Тестирование API модуля курсов в Postman проводится для эндпоинтов, порождённых универсальным роутером: GET списка курсов, GET курса по id, при необходимости POST/PUT (от имени грумера или админа), а также для связанных ресурсов — расписания, преподавателей, записей на курсы.

**Эндпоинты и сценарии.** GET `http://localhost:3001/api/courses` — получение списка курсов с пагинацией (query-параметры page, limit) и опциональными фильтрами. Ожидаемый ответ: JSON с полями success: true, data: массив объектов курсов (id, name, level, format, duration, price, description, image, created_at, updated_at, created_by), pagination: { page, limit, total, pages }. Для гостя и клиента запрос без заголовка Authorization или с валидным JWT клиента должен возвращать 200 и список. GET `http://localhost:3001/api/courses/:id` — получение одного курса по идентификатору; ответ success, data: объект курса. GET `http://localhost:3001/api/course_schedules?course_id=1&limit=10` — расписание по курсу; GET `http://localhost:3001/api/course_instructors?course_id=1` — преподаватели курса; GET `http://localhost:3001/api/reviews?course_id=1&type=course` — отзывы по курсу. Для авторизованного клиента проверяется создание записи на курс: POST `http://localhost:3001/api/course_bookings` с телом { course_id, course_schedule_id, master_id (опционально), notes } и заголовком Authorization: Bearer <JWT клиента>; в ответе — 201 и созданная запись с user_id текущего пользователя.

**Проверка прав и ошибок.** Запросы без токена к защищённым методам (если бы они требовали авторизации) проверяются отдельно; для read курсов доступ открыт гостям, поэтому 403 не ожидается. POST/PUT к `/api/courses` без токена или с токеном клиента должны возвращать 403 «Нет прав на создание». При неверном id в GET `/api/courses/99999` ожидается 404 «Запись не найдена». В Postman удобно сохранить коллекцию «MARS GROOM — Курсы» с перечисленными запросами и переменными окружения (base_url, token) для повторяемого тестирования после каждого изменения бэкенда или миграций.

**Пошаговый сценарий тестирования.** Сначала выполняется GET /api/courses без заголовка Authorization — ожидается статус 200 и массив курсов в поле data. Затем GET /api/courses/1 — одна запись с полями id, name, level, format, duration, price, description, image. Для проверки фильтрации: GET /api/course_schedules?course_id=1 — должны вернуться только записи расписания для курса с id=1. Аналогично для course_instructors и reviews с параметром course_id. Для проверки создания записи на курс необходимо получить JWT клиента (POST /api/auth/login с email и паролем тестового клиента из сидов), подставить его в Authorization и отправить POST /api/course_bookings с телом { course_id: 1, course_schedule_id: 1, notes: "Тест" }. Ответ должен содержать статус 201 и объект с полями id, user_id (равный id текущего пользователя), course_id, course_schedule_id, status и т.д. Проверка поиска (если реализован эндпоинт GET /api/courses/search?q=...) позволяет убедиться в работе поискового метода универсального контроллера. Документирование результатов тестирования (скриншоты ответов, примеры тел запросов) удобно хранить в той же коллекции Postman или в отдельном отчёте для приложения к дипломной работе.

## 4. Разработка универсального хука (применение в модуле курсов)

Универсальный хук useEntity обеспечивает единообразную работу фронтенда с любым CRUD-ресурсом API, в том числе с курсами и всеми связанными сущностями модуля курсов (расписание, преподаватели, отзывы, модули, запись на курс).

**Использование в страницах курсов.** На CourseDetailPage хук вызывается несколько раз: useEntity<CourseRow>('courses', { fetchListOnMount: false, id: courseId, fetchItemOnMount: !!courseId, enabled: !!courseId }) — загрузка одного курса по id из URL; useEntity<ScheduleRow>('course_schedules', { fetchListOnMount: false }) с последующим refetchList({ course_id: courseId, limit: 20 }) для расписания по курсу; useEntity<ReviewRow>('reviews', { fetchListOnMount: false }) и refetchList({ course_id: courseId, type: 'course', limit: 50 }) для отзывов; useEntity для course_instructors, masters, course_modules с параметрами listParams { course_id: courseId }. Таким образом, одна и та же абстракция используется для загрузки одной записи (курс) и списков с фильтрами (расписание, отзывы, преподаватели, модули), без дублирования логики запросов и обработки ошибок. На странице списка курсов (если она переведена на API) применяется useEntity<CourseRow>('courses', { fetchListOnMount: true, listParams: { limit: 50 } }), что даёт list, loadingList, loadingListError и возможность отобразить карточки или индикатор загрузки/ошибку. Для записи на курс форма может вызывать create из useEntity('course_bookings', …) с телом { course_id, course_schedule_id, notes }; хук сам отправит POST и обновит локальное состояние при успехе.

**Преимущества для модуля курсов.** Типизация (CourseRow, ScheduleRow и т.д.) даёт подсказки и проверку полей; единый формат ответов API (success, data, pagination) обрабатывается внутри хука; состояния loadingItem, loadingItemError, creating, createError позволяют показывать скелетоны и сообщения об ошибках без ручного управления флагами. За счёт этого фронтенд модуля курсов остаётся простым и консистентным при добавлении новых полей или сущностей (например, тесты курсов, прогресс), достаточно подключить ещё один useEntity с нужным именем ресурса и параметрами.

# ЧАСТЬ 2. МОДУЛЬ «УСЛУГИ»

## 1. Разработка фронтенда модуля услуг

Фронтенд модуля услуг реализован на React (TypeScript), Motion, React Router и Tailwind CSS и отвечает за витрину груминга: прайс-лист, карточки услуг, детальные страницы услуг с мастерами и запись на услугу.

**Страницы и компоненты.** Главная страница раздела услуг (ServicesMainPage) — лендинг с hero-блоком в синей фирменной гамме (#4A90E2, #9EC3EF), ссылками на список услуг и запись. Страница «Прайс-лист» / список услуг (ServicesListPage) загружает список через useEntity<ServiceRow>('services', { fetchListOnMount: true, listParams: { limit: 100 } }). Отображаются фильтры по категориям (все / собаки / кошки / другое) и поиск по названию, описанию и породе; карточки услуг содержат изображение (service.image или заглушка), цену (price_range или price), длительность, описание и кнопки «Подробнее» (переход на /services/:id) и «Записаться» (переход на /book/service/:id). Состояния загрузки и ошибки (loadingList, loadingListError) обрабатываются с показом Loader2 или текста ошибки. Страница детального просмотра услуги (ServiceDetailPage) получает одну услугу по id из useParams и связанные данные: useEntity('services', { id: serviceId, fetchItemOnMount: true }), useEntity('masters'), useEntity('master_services', { listParams: { service_id: serviceId } }). На странице выводятся название, цена, длительность, описание, что входит в услугу, противопоказания, подготовка, рекомендуемые средства, блок «до/после» (фото), список мастеров, оказывающих эту услугу (с кнопками «Подробнее» и «Записаться к мастеру»), похожие услуги (по категории/типу из того же списка services) и форма обратной связи. Компоненты FavoriteButton (для favorite_services), ContactForm, ServicesHeader, ServicesFooter используются повторно. Страница записи на услугу (ServiceBookingPage) содержит форму выбора услуги, мастера, питомца, даты и времени (слоты с API service_bookings/slots), контактов и отправляет POST в service_bookings.

**Технологии и приёмы.** Типы ServiceRow, MasterFromApi и др. описывают структуру данных с бэкенда; useMemo применяется для вычисления похожих услуг, списка мастеров по услуге и производных полей (whatIncluded, contraindications, preparation, products, beforeAfter) на основе category и type услуги. Стили выдержаны в синей теме услуг, адаптивная вёрстка и тёмная тема учтены. Таким образом, фронтенд модуля услуг обеспечивает каталог, детальную карточку услуги с мастерами и запись на услугу, с единой точкой работы с API через useEntity.

**Детали реализации модуля услуг.** Маршруты: /services (главная), /services/list (прайс-лист), /services/:id (детальная страница), /book/service и /book/service/:id (запись). На странице списка фильтрация по категории (dogs, cats, other) и поиск выполняются на клиенте по уже загруженному списку услуг, что уменьшает число запросов к серверу при переключении вкладок. На детальной странице список мастеров, оказывающих услугу, получается пересечением списка master_services (с фильтром service_id) и списка masters; для каждого мастера отображаются фото, имя, рейтинг и кнопки «Подробнее» (переход на /masters/:id) и «Записаться к этому мастеру». Блок «похожие услуги» формируется из того же списка services по совпадению category или type. Форма записи на услугу (ServiceBookingPage) может использовать эндпоинт слотов (getSlots или GET /api/service_bookings/slots) для выбора даты и времени и затем create из useEntity('service_bookings') для отправки заявки. Обработка пустых состояний (нет мастеров, нет похожих услуг) и ошибок сети выполняется через условия на loadingItem, loadingItemError и длину массивов. В итоге фронтенд модуля услуг даёт полноценный пользовательский путь от выбора услуги до создания записи с минимальным дублированием кода благодаря useEntity.

## 2. Разработка универсального контроллера и создание БД (модуль услуг)

Модуль услуг опирается на те же универсальный контроллер и фабрику роутеров; таблицы услуг и связанных сущностей описаны в миграциях и обслуживаются CRUD-методами с проверкой прав.

**Таблицы БД, относящиеся к услугам.** В схеме созданы: `services` (id, name, category, type, price, duration, description, image, breed, price_range, timestamps, created_by); `master_services` (id, master_id, service_id, timestamps, created_by) с уникальной парой (master_id, service_id); `service_bookings` (id, user_id, service_id, master_id, pet_id, scheduled_at, status, contact_method, notes, timestamps, created_by). Дополнительно для визитов и отзывов: `visits` (связь с service_booking_id, service_id, master_id), `reviews` (service_id, course_id, master_id), `favorite_services` (user_id, service_id). Категория и тип услуги (dogs, cats, other; grooming, bathing, nail, extra) задаются при создании и используются на фронте для фильтрации и отображения. Универсальный контроллер для таблицы services вызывается через createEntityRouter('services', { guest: { read: true }, client: { read: true }, groomer: { read: true, create: true, update: true, delete: false } }), то есть чтение доступно всем, создание и редактирование — грумерам. Роутеры master_services, service_bookings создаются той же фабрикой; для service_bookings обычно задаётся ownerField: 'user_id', чтобы клиент видел и создавал только свои записи.

**Общая логика контроллера.** getAll принимает query (page, limit и любые поля для фильтрации), проверяет права, строит запрос к БД через Knex, применяет фильтры, сортировку по created_at, пагинацию и возвращает { success, data, pagination }. getById загружает запись по id, проверяет право на чтение и возвращает её или 404/403. create добавляет запись из req.body, при необходимости подставляет ownerField из req.user.id. update и delete проверяют право на изменение/удаление (в том числе updateOwn/deleteOwn по владельцу). Для модуля услуг это даёт готовые эндпоинты GET/POST/PUT/DELETE для services, master_services, service_bookings без написания отдельной логики под каждую таблицу. Сиды заполняют services тестовыми услугами (названия, категории, цены, изображения), master_services связями мастер–услуга и service_bookings примерами записей для проверки фронтенда и тестов в Postman.

## 3. Тестирование в Postman (модуль услуг)

В Postman проверяются эндпоинты API для услуг, связей мастер–услуга и записей на услуги, включая пагинацию, фильтры и разграничение прав.

**Эндпоинты и сценарии.** GET `http://localhost:3001/api/services` — список услуг; с query `?page=1&limit=20` или `?category=dogs` ожидается ответ с массивом data и pagination. GET `http://localhost:3001/api/services/:id` — одна услуга по id; при несуществующем id — 404. GET `http://localhost:3001/api/master_services?service_id=1` — мастера по услуге; GET `http://localhost:3001/api/masters` — список мастеров для проверки связей. GET `http://localhost:3001/api/service_bookings/slots?date=2026-02-15&service_id=1` — слоты для записи (если реализован отдельный обработчик слотов). Для авторизованного клиента: POST `http://localhost:3001/api/service_bookings` с телом { service_id, master_id, pet_id (опционально), scheduled_at (ISO), contact_method, notes }; заголовок Authorization: Bearer <JWT>; ожидается 201 и объект записи с user_id текущего пользователя. GET `http://localhost:3001/api/service_bookings` с тем же JWT должен возвращать только записи этого пользователя (благодаря ownerField и readOwn). Проверка прав: POST к `/api/services` без токена или с ролью client — 403; с токеном groomer или admin — 201 при валидном теле.

**Проверка ошибок и граничных случаев.** Невалидный id в GET (например, /api/services/0 или строковый id) — 400 или 404 в зависимости от валидации. Создание service_booking без обязательных полей (service_id, scheduled_at) — 500 или валидационная ошибка. Таким образом, тестирование в Postman для модуля услуг покрывает чтение каталога и деталей услуги, получение мастеров по услуге, слотов и записей, а также создание записей и проверку прав доступа.

**Пошаговый сценарий для модуля услуг.** Сначала GET /api/services — проверка, что возвращается список с полями name, category, type, price, duration, image. Затем GET /api/services/1 — детали первой услуги. GET /api/master_services?service_id=1 — список связей; по master_id из ответа можно запросить GET /api/masters/:id для проверки данных мастера. Для теста записи на услугу: авторизация клиента (POST /api/auth/login), затем POST /api/service_bookings с телом { service_id: 1, master_id: 1, scheduled_at: "2026-02-20T10:00:00.000Z", contact_method: "по звонку" } и заголовком Authorization. В ответе — 201 и объект с id, user_id (текущий пользователь), service_id, status: "pending". Затем GET /api/service_bookings с тем же токеном — в списке должна быть только что созданная запись. Для проверки прав: POST /api/services с телом нового курса без токена или с токеном клиента — ожидается 403. Создание коллекции «MARS GROOM — Услуги» в Postman с папками «Чтение», «Запись», «Права» упрощает регрессионное тестирование при изменениях в API.

## 4. Разработка универсального хука (применение в модуле услуг)

В модуле услуг useEntity используется для загрузки списка услуг, одной услуги по id, списка мастеров, связей master_services и при необходимости списка записей пользователя и создания новой записи.

**Использование на страницах услуг.** ServicesListPage: useEntity<ServiceRow>('services', { fetchListOnMount: true, listParams: { limit: 100 } }); из хука берутся list (массив услуг), loadingList, loadingListError; список фильтруется на клиенте по категории и поисковой строке. ServiceDetailPage: useEntity<ServiceFromApi>('services', { fetchListOnMount: true, listParams: { limit: 100 }, id: serviceId, fetchItemOnMount: !!serviceId, enabled: !!serviceId }) — одновременно загружаются и список (для похожих услуг), и одна запись (item) по id; useEntity('masters', { fetchListOnMount: true, listParams: { limit: 50 } }); useEntity('master_services', { fetchListOnMount: true, listParams: { limit: 200, service_id: serviceId } }). По listParams с service_id бэкенд возвращает только связи по данной услуге, что соответствует фильтрации в universalController (query-параметры передаются в where). На странице записи (ServiceBookingPage) для отправки заявки может использоваться create из useEntity('service_bookings', …), что избавляет от ручного вызова api.post и обновления состояния. Состояния creating, createError позволяют показывать индикатор отправки и сообщения об ошибках.

**Преимущества для модуля услуг.** Один и тот же хук обслуживает и публичный каталог (services), и детальную страницу (item + списки masters, master_services), и создание записей (service_bookings), сохраняя единообразие типов и обработки ошибок. При добавлении новых полей в таблицу services или новых сущностей (например, пакетов услуг) достаточно обновить типы на фронте и при необходимости добавить новый роутер на бэкенде; вызов useEntity с новым именем ресурса даёт полный CRUD-интерфейс без дублирования кода запросов.

# ЧАСТЬ 3. ОБЩАЯ ЧАСТЬ

## 1. Общая часть: разработка фронтенда

Общая архитектура фронтенда едина для обоих модулей и всего приложения: React 18, TypeScript, Vite, React Router v7, библиотека Motion (framer-motion), Tailwind CSS, Radix UI для части компонентов. Роутинг вынесен в корневой компонент (App.tsx): маршруты /courses, /courses/list, /courses/:id, /book/course и т.д. для курсов; /services, /services/list, /services/:id, /book/service для услуг. Общие компоненты — Header/Footer (в вариантах CoursesHeader, ServicesHeader и общий Footer), ContactForm, FavoriteButton, GroomerPath — используются и в модуле курсов, и в модуле услуг. API-клиент (api.get, api.post, api.put, api.delete) один на всё приложение; базовый URL и JWT хранятся в конфигурации и localStorage. Таким образом, модули курсов и услуг не дублируют инфраструктуру, а отличаются наборами страниц, сущностями useEntity и фирменными цветами (зелёный для курсов, синий для услуг).

---

## 2. Общая часть: универсальный контроллер и БД

Универсальный контроллер (universalController.js) и фабрика роутеров (createEntityRouter) общие для всех 44+ сущностей проекта: один и тот же код реализует getAll, getById, create, update, delete, search с проверкой прав по ролям (guest, client, groomer, admin) и опциями ownerField, excludeFields, allowGuestCreate. База данных создаётся одной миграцией (Knex): таблицы users, masters, services, courses, course_schedules, master_services, course_instructors, pets, service_bookings, course_bookings и остальные с единообразными полями id, created_at, updated_at, created_by. Модули курсов и услуг не имеют отдельных контроллеров — они используют общий механизм с разными именами таблиц и настройками прав в createEntityRouter. Это сокращает объём кода бэкенда и обеспечивает одинаковое поведение API (формат ответов, пагинация, фильтрация по query) для курсов, услуг и остальных разделов.

## 3. Общая часть: тестирование в Postman

Подход к тестированию в Postman общий: для любого ресурса, подключённого через createEntityRouter, проверяются GET списка (с page, limit и фильтрами), GET по id, при необходимости POST/PUT/DELETE с учётом роли. Формат запросов и ответов единый (JSON, success, data, error, pagination). Для авторизованных сценариев используется переменная token и заголовок Authorization: Bearer {{token}}. Коллекции можно группировать по модулям («Курсы», «Услуги», «Пользователи» и т.д.), но сами запросы строятся по одному шаблону: метод, URL, опциональное тело, проверка статуса и тела ответа. Общая часть тестирования также включает проверку health (GET /api/health), авторизации (POST /api/auth/login) и при необходимости CORS и обработку ошибок (404, 403, 500).

---

## 4. Общая часть: универсальный хук useEntity

Универсальный хук useEntity реализован один раз в проекте (hooks/useEntity.ts) и принимает имя сущности (строку, соответствующую пути API: courses, services, course_schedules, service_bookings и т.д.) и опции (fetchListOnMount, listParams, id, fetchItemOnMount, enabled). Он возвращает list, item, pagination, refetchList, refetchItem, create, update, remove и все флаги загрузки и ошибок (loadingList, loadingItem, creating, updating, deleting и соответствующие Error). Внутри хук использует единый api-клиент и формирует запросы GET/POST/PUT/DELETE к базовому пути /api/{entityName}. Модули курсов и услуг не реализуют собственных хуков для работы с API — они только вызывают useEntity с разными именами ресурсов и типами (CourseRow, ServiceRow и т.д.). Общая часть включает также типы ApiListResponse, ApiItemResponse, ApiErrorResponse и соглашение о формате ответов бэкенда (success, data, pagination), что обеспечивает совместимость фронтенда и бэкенда для любых текущих и будущих сущностей приложения.

**Внутренняя реализация useEntity.** Состояния list, item, pagination хранятся в useState; запросы выполняются в useCallback (refetchList, refetchItem, create, update, remove), чтобы ссылки на функции не менялись без необходимости. В refetchList параметры listParams и переданные params объединяются в URLSearchParams и подставляются в GET-запрос; ответ разбирается на data и pagination и записывается в состояние. В create и update при успешном ответе локальный list и item обновляются (добавление новой записи в начало списка или замена по id), что обеспечивает актуальный UI без дополнительного refetchList. В useEffect при монтировании при включённых fetchListOnMount и fetchItemOnMount вызываются refetchList() и refetchItem(id). Зависимости в useEffect ограничены, чтобы не вызывать лишние запросы при каждом рендере. Таким образом, общая реализация хука обеспечивает и модулю курсов, и модулю услуг единый способ загрузки данных, мутаций и отображения состояний загрузки и ошибок без дублирования кода на каждой странице.
