# ПУНКТ 1. РАЗРАБОТКА ФРОНТЕНДА

## 1.1. Модуль курсов — разработка фронтенда

Фронтенд модуля курсов веб-приложения MARS GROOM реализован на React с использованием TypeScript, сборщика Vite, библиотеки Motion для анимаций, React Router для маршрутизации и Tailwind CSS для стилей. Модуль курсов отвечает за витрину образовательных программ по грумингу: от главной страницы раздела до детального просмотра курса, расписания стартов и записи на курс.

**Страницы модуля курсов.** Главная точка входа — страница «Курсы» (CoursesMainPage): лендинг с hero-блоком в зелёной фирменной цветовой гамме (#40AB40, #89E689), фоновым изображением, призывом к действию «Список всех курсов» и «Запись на курс». Ниже размещены блок «Путь грумера» (GroomerPath), секция с отзывами по курсам в виде карусели, блок теста и форма обратной связи (ContactForm). Страница списка курсов (CoursesPage) выводит карточки курсов; данные подгружаются с бэкенда через универсальный хук useEntity для сущности `courses` с параметрами fetchListOnMount: true и listParams: { limit: 50 }. Каждая карточка отображает название курса, уровень (beginner/advanced), формат (online, offline, hybrid), длительность, цену, изображение и кнопки перехода к детальной странице и записи. Страница детального просмотра курса (CourseDetailPage) показывает полное описание одного курса: название, уровень, формат, длительность, цену, описание, изображение, программу (модули и контент через course_modules и course_content), преподавателя (через связь course_instructors и masters), отзывы и кнопки «В избранное» (favorite_courses) и «Записаться». Для загрузки данных используются несколько экземпляров useEntity: курсы (courses) по id с fetchItemOnMount, расписание (course_schedules) с listParams { course_id }, отзывы (reviews) с фильтром course_id и type: 'course', преподаватели (course_instructors), мастера (masters), модули (course_modules). Это обеспечивает единообразную работу с API и актуальные данные из БД без дублирования логики запросов. Страница расписания (CourseSchedulePage) отображает ближайшие даты стартов курсов с фильтром по формату (все / онлайн / очно / гибрид); данные о курсах и слотах загружаются из API course_schedules с привязкой к курсам. Страница записи на курс (CourseBookingPage) содержит форму выбора курса, даты и времени из расписания, контактных данных и отправляет заявку через create из useEntity('course_bookings').

**Технологии и приёмы в модуле курсов.** На фронтенде используются хуки useState, useEffect, useMemo для состояния и производных данных; типизация TypeScript для сущностей (CourseRow, ScheduleRow, ReviewRow, MasterRow и т.д.); единый API-клиент (api.get, api.post) с базовым URL и JWT в заголовке Authorization; обработка состояний загрузки (loadingList, loadingItem) и ошибок (loadingListError, loadingItemError) с отображением индикаторов (Loader2) и сообщений пользователю. Стилизация выдержана в зелёной цветовой схеме курсов, адаптивная вёрстка и тёмная тема (dark:) учтены. Маршруты модуля курсов регистрируются в App.tsx: /courses (главная), /courses/list (список), /courses/schedule (расписание), /courses/:id (детальная страница), /book/course (запись). Дополнительно задействованы компоненты CoursesGallery, CoursesHeader, CoursesFooter, FavoriteButton для избранных курсов (favorite_courses), общие ContactForm и GroomerPath. Блок программы на детальной странице строится по данным course_modules и course_content: модули отображаются в виде аккордеона или списка с длительностью и типом контента (видео, текст). Анимации появления блоков (motion.div с initial, animate, transition) улучшают восприятие без перегрузки контентом. Итоговая структура фронтенда модуля курсов позволяет пользователю познакомиться с программой обучения, выбрать курс и дату старта и оставить заявку на запись, при этом вся бизнес-логика загрузки и обновления данных сосредоточена в универсальном хуке useEntity и общем API-клиенте.

## 1.2. Модуль услуг — разработка фронтенда

Фронтенд модуля услуг реализован на React (TypeScript), Motion, React Router и Tailwind CSS и отвечает за витрину груминга: прайс-лист, карточки услуг, детальные страницы услуг с мастерами, страницу мастера и запись на услугу.

**Страницы модуля услуг.** Главная страница раздела услуг (ServicesMainPage) — лендинг с hero-блоком в синей фирменной гамме (#4A90E2, #9EC3EF), ссылками на список услуг и запись. Страница «Прайс-лист» / список услуг (ServicesListPage) загружает список через useEntity<ServiceRow>('services', { fetchListOnMount: true, listParams: { limit: 100 } }). Отображаются фильтры по категориям (все / собаки / кошки / другое) и поиск по названию, описанию и породе; карточки услуг содержат изображение (service.image или заглушка), цену (price_range или price), длительность, описание и кнопки «Подробнее» (переход на /services/:id) и «Записаться» (переход на /book/service/:id). Состояния загрузки и ошибки (loadingList, loadingListError) обрабатываются с показом Loader2 или текста ошибки. Страница детального просмотра услуги (ServiceDetailPage) получает одну услугу по id из useParams и связанные данные: useEntity('services', { id: serviceId, fetchItemOnMount: true }), useEntity('masters'), useEntity('master_services', { listParams: { service_id: serviceId } }). На странице выводятся название, цена, длительность, описание, блок «что входит», противопоказания, подготовка, рекомендуемые средства, блок «до/после» (фото), список мастеров, оказывающих эту услугу (с кнопками «Подробнее» — переход на /masters/:id — и «Записаться к этому мастеру»), похожие услуги (по категории/типу из списка services) и форма обратной связи. Страница мастера (MasterDetailPage) загружает данные мастера и портфолио через useEntity('masters', { id, fetchItemOnMount }) и useEntity('groomer_portfolio', { listParams: { master_id } }). Страница записи на услугу (ServiceBookingPage) содержит форму выбора услуги, мастера, питомца, даты и времени (слоты через API service_bookings/slots), контактов и отправляет POST в service_bookings через useEntity или api. Дополнительно в модуле используются компоненты FavoriteButton (favorite_services), ContactForm, ServicesHeader, ServicesFooter, общие для приложения.

**Технологии и приёмы в модуле услуг.** Типы ServiceRow, MasterFromApi и др. описывают структуру данных с бэкенда; useMemo применяется для вычисления похожих услуг, списка мастеров по услуге и производных полей (whatIncluded, contraindications, preparation, products, beforeAfter) на основе category и type услуги. Маршруты: /services (главная), /services/list (прайс-лист), /services/:id (детальная страница), /masters/:id (страница мастера), /book/service и /book/service/:id (запись). На странице списка фильтрация по категории (dogs, cats, other) и поиск выполняются на клиенте по уже загруженному списку услуг. На детальной странице список мастеров формируется по пересечению master_services (с фильтром service_id) и masters; для каждого мастера отображаются фото, имя, рейтинг и кнопки «Подробнее» и «Записаться». Форма записи использует эндпоинт слотов (api.getSlots или GET /api/service_bookings/slots) для выбора даты и времени и create из useEntity('service_bookings') для отправки заявки. Обработка пустых состояний и ошибок выполняется через условия на loadingItem, loadingListError и длину массивов. Стили выдержаны в синей теме услуг, адаптивная вёрстка и тёмная тема учтены. Таким образом, фронтенд модуля услуг обеспечивает каталог, детальную карточку услуги с мастерами, страницу мастера и запись на услугу с единой точкой работы с API через useEntity.

## 1.3. Общая часть — разработка фронтенда

Общая архитектура фронтенда едина для модулей курсов, услуг и всего приложения. Используются React 18, TypeScript, Vite, React Router v7, библиотека Motion (framer-motion), Tailwind CSS, часть компонентов на Radix UI. Роутинг вынесен в корневой компонент App.tsx: маршруты для курсов (/courses, /courses/list, /courses/schedule, /courses/:id, /book/course) и для услуг (/services, /services/list, /services/:id, /masters/:id, /book/service, /book/service/:id) регистрируются рядом с маршрутами лендинга, личного кабинета и администратора. Общие компоненты — Header и Footer в вариантах CoursesHeader/CoursesFooter (для раздела курсов) и ServicesHeader/ServicesFooter (для раздела услуг), а также единые ContactForm, FavoriteButton, GroomerPath — используются в обоих модулях и на лендинге. API-клиент (api.get, api.post, api.put, api.delete) один на всё приложение; базовый URL задаётся через VITE_API_URL или по умолчанию http://localhost:3001/api, JWT хранится в localStorage (mars_groom_token) и подставляется в заголовок Authorization при каждом запросе. Состояния загрузки и ошибок обрабатываются единообразно: индикатор (Loader2) при loadingList/loadingItem, текст ошибки при loadingListError/loadingItemError. Модули курсов и услуг не дублируют инфраструктуру: общие хуки (useEntity), общий API-клиент, общие типы ответов (ApiListResponse, ApiItemResponse, ApiErrorResponse). Отличие модулей — набор страниц, сущности, с которыми работает useEntity (courses, course_schedules, course_bookings и т.д. против services, master_services, service_bookings, masters), и фирменные цвета (зелёный для курсов, синий для услуг). Темная тема и адаптивная вёрстка применяются глобально через Tailwind-классы dark: и брейкпоинты. Таким образом, общая часть фронтенда обеспечивает согласованность технологий, маршрутизации, работы с API и стилей для обоих модулей и всего веб-приложения.

# ПУНКТ 2. РАЗРАБОТКА УНИВЕРСАЛЬНОГО КОНТРОЛЛЕРА — СОЗДАНИЕ БД

## 2.1. Модуль курсов — универсальный контроллер и БД

Универсальный CRUD-контроллер (universalController.js) и фабрика роутеров createEntityRouter используются для всех сущностей приложения, в том числе для модуля курсов. Роутеры курсов и связанных с ними сущностей создаются одним вызовом createEntityRouter с именем таблицы и настройками прав, что даёт согласованное API и разграничение доступа по ролям (guest, client, groomer, admin).

**Таблицы БД, относящиеся к курсам.** В миграции Knex создаются таблицы: courses (id, name, level, format, duration, price, description, image, created_at, updated_at, created_by); course_schedules (id, course_id, start_date, start_time, spots, timestamps, created_by); course_instructors (id, course_id, master_id, timestamps, created_by) с уникальной парой (course_id, master_id); course_bookings (id, user_id, course_id, course_schedule_id, master_id, status, notes, timestamps, created_by); course_modules (id, course_id, title, description, sort_order); course_content (id, module_id, title, type, content, file_path, duration_minutes); user_course_progress; course_quiz_questions, course_quiz_options, course_quiz_answers; favorite_courses (user_id, course_id). Связи заданы через внешние ключи (references) на users, masters, courses, course_schedules с соответствующими onDelete (CASCADE, SET NULL, RESTRICT). Во всех таблицах присутствуют поля created_at, updated_at, created_by (nullable), что поддерживается универсальным контроллером при создании и обновлении записей (created_by при создании подставляется из req.user.id при наличии авторизации).

**Настройка прав для курсов.** Роутер courses создаётся вызовом createEntityRouter('courses', { guest: { read: true, create: false, update: false, delete: false }, client: { read: true, create: false, update: false, delete: false }, groomer: { read: true, create: true, update: true, delete: false } }). Таким образом, гости и клиенты могут только читать список и детали курсов; создание и изменение доступны грумерам и админам; удаление отключено для всех. Универсальный контроллер по таблице courses реализует методы getAll (список с пагинацией page, limit и фильтрами по query-параметрам), getById (одна запись по id), create, update, delete; перед каждой операцией выполняется проверка прав checkPermission по роли пользователя (req.user задаётся middleware optionalAuth). Для course_schedules, course_instructors, course_bookings, course_modules, course_content и других сущностей курсов используются отдельные роутеры с той же фабрикой и своими настройками прав. В частности, course_bookings создаётся с ownerField: 'user_id', что позволяет клиенту читать и обновлять только свои записи (readOwn, updateOwn) и создавать новые (create: true); при создании записи user_id подставляется из req.user.id. Создание БД выполняется миграциями (knex migrate:latest); тестовые данные для курсов, расписания, преподавателей, записей и модулей заносятся сидами (seeds), что позволяет сразу проверять API и фронтенд модуля курсов без ручного заполнения таблиц.

## 2.2. Модуль услуг — универсальный контроллер и БД

Модуль услуг опирается на тот же универсальный контроллер и фабрику createEntityRouter; таблицы услуг и связанных сущностей описаны в той же миграции и обслуживаются CRUD-методами с проверкой прав.

**Таблицы БД, относящиеся к услугам.** В схеме созданы: services (id, name, category, type, price, duration, description, image, breed, price_range, timestamps, created_by); master_services (id, master_id, service_id, timestamps, created_by) с уникальной парой (master_id, service_id); service_bookings (id, user_id, service_id, master_id, pet_id, scheduled_at, status, contact_method, notes, timestamps, created_by). Дополнительно для визитов и отзывов используются таблицы visits (связь с service_booking_id, service_id, master_id), reviews (service_id, course_id, master_id, type), favorite_services (user_id, service_id). Категория и тип услуги (category: dogs, cats, other; type: grooming, bathing, nail, extra) задаются при создании и используются на фронте для фильтрации и отображения. Универсальный контроллер для таблицы services подключается через createEntityRouter('services', { guest: { read: true }, client: { read: true }, groomer: { read: true, create: true, update: true, delete: false } }), то есть чтение доступно всем, создание и редактирование — грумерам и админам. Роутеры master_services и service_bookings создаются той же фабрикой; для service_bookings задаётся ownerField: 'user_id', чтобы клиент видел и создавал только свои записи. Кроме того, для service_bookings реализован отдельный эндпоинт GET /slots (в файле service_bookings.js) для получения доступных слотов по дате, master_id и service_id с учётом длительности услуги и уже занятых записей; создание и обновление записей проходят через validateSlot (проверка, что scheduled_at попадает в разрешённое окно). Общая логика getAll: приём query (page, limit и любые поля для фильтрации), проверка прав (в том числе readOwn с подстановкой filters[ownerField] = user.id), построение запроса к БД через Knex, применение фильтров, сортировка по created_at, пагинация, возврат { success, data, pagination }. Сиды заполняют services тестовыми услугами (названия, категории, цены, изображения), master_services — связями мастер–услуга, service_bookings — примерами записей для проверки фронтенда и тестов в Postman.

## 2.3. Общая часть — универсальный контроллер и БД

Универсальный контроллер (universalController.js) и фабрика роутеров (createEntityRouter в _createEntityRouter.js) общие для всех сущностей проекта (в том числе 44+ таблиц): один и тот же код реализует getAll, getById, create, update, delete и search с проверкой прав по ролям (guest, client, groomer, admin) и опциями ownerField, excludeFields, allowGuestCreate, searchField. База данных создаётся одной миграцией Knex: таблицы users, masters, services, courses, course_schedules, master_services, course_instructors, pets, service_bookings, course_bookings и остальные (reviews, visits, favorite_services, favorite_courses, course_modules, course_content и т.д.) с единообразными полями id, created_at, updated_at, created_by. Контроллер не привязан к конкретному модулю: он принимает имя таблицы (tableName), конфигурацию прав (rolesConfig) и опции (options); роутеры для курсов и услуг отличаются только этими параметрами. Метод getAll извлекает из req.query параметры page, limit и все остальные как фильтры; если у роли нет read, но есть readOwn и передан user.id, в фильтры добавляется ownerField: user.id. Методы create, update, delete проверяют соответствующие права (create, update, delete и при необходимости updateOwn, deleteOwn по владельцу записи). Формат ответов единый: при успехе — { success: true, data: ... }, при списке — дополнительно pagination: { page, limit, total, pages }; при ошибке — { success: false, error: "..." }. Модули курсов и услуг не имеют отдельных контроллеров — они используют общий механизм с разными именами таблиц и настройками прав. Это сокращает объём кода бэкенда и обеспечивает одинаковое поведение API (формат ответов, пагинация, фильтрация по query) для курсов, услуг и остальных разделов приложения.

# ПУНКТ 3. ТЕСТИРОВАНИЕ В POSTMAN

## 3.1. Модуль курсов — тестирование в Postman

Тестирование API модуля курсов в Postman проводится для эндпоинтов, порождённых универсальным роутером: GET списка курсов, GET курса по id, при необходимости POST/PUT от имени грумера или админа, а также для связанных ресурсов — расписания, преподавателей, записей на курсы.

**Эндпоинты и сценарии.** GET http://localhost:3001/api/courses — получение списка курсов с пагинацией (query-параметры page, limit) и опциональными фильтрами. Ожидаемый ответ: JSON с полями success: true, data: массив объектов курсов (id, name, level, format, duration, price, description, image, created_at, updated_at, created_by), pagination: { page, limit, total, pages }. Для гостя и клиента запрос без заголовка Authorization или с валидным JWT клиента должен возвращать 200 и список. GET http://localhost:3001/api/courses/:id — получение одного курса по идентификатору; ответ success, data: объект курса. GET http://localhost:3001/api/course_schedules?course_id=1&limit=10 — расписание по курсу; GET http://localhost:3001/api/course_instructors?course_id=1 — преподаватели курса; GET http://localhost:3001/api/reviews?course_id=1&type=course — отзывы по курсу. Для авторизованного клиента проверяется создание записи на курс: POST http://localhost:3001/api/course_bookings с телом { course_id, course_schedule_id, master_id (опционально), notes } и заголовком Authorization: Bearer <JWT клиента>; в ответе — 201 и созданная запись с user_id текущего пользователя. Проверка прав: POST или PUT к /api/courses без токена или с токеном клиента должны возвращать 403 «Нет прав на создание» или «Нет прав на обновление». При неверном id в GET /api/courses/99999 ожидается 404 «Запись не найдена».

**Пошаговый сценарий для модуля курсов.** Сначала выполняется GET /api/courses без заголовка Authorization — ожидается статус 200 и массив курсов в поле data. Затем GET /api/courses/1 — одна запись с полями id, name, level, format, duration, price, description, image. Для проверки фильтрации: GET /api/course_schedules?course_id=1 — должны вернуться только записи расписания для курса с id=1. Аналогично для course_instructors и reviews с параметром course_id. Для проверки создания записи на курс необходимо получить JWT клиента (POST /api/auth/login с email и паролем тестового клиента из сидов), подставить его в Authorization и отправить POST /api/course_bookings с телом { course_id: 1, course_schedule_id: 1, notes: "Тест" }. Ответ должен содержать статус 201 и объект с полями id, user_id (равный id текущего пользователя), course_id, course_schedule_id, status и т.д. Документирование результатов (скриншоты ответов, примеры тел запросов) удобно хранить в коллекции Postman «MARS GROOM — Курсы» с переменными окружения base_url, token для повторяемого тестирования после изменений бэкенда или миграций.

## 3.2. Модуль услуг — тестирование в Postman

В Postman проверяются эндпоинты API для услуг, связей мастер–услуга и записей на услуги, включая пагинацию, фильтры, слоты и разграничение прав.

**Эндпоинты и сценарии.** GET http://localhost:3001/api/services — список услуг; с query ?page=1&limit=20 или ?category=dogs ожидается ответ с массивом data и pagination. GET http://localhost:3001/api/services/:id — одна услуга по id; при несуществующем id — 404. GET http://localhost:3001/api/master_services?service_id=1 — мастера по услуге; GET http://localhost:3001/api/masters — список мастеров для проверки связей. GET http://localhost:3001/api/service_bookings/slots?date=2026-02-15&service_id=1&master_id=1 — слоты для записи на выбранную дату с учётом длительности услуги и занятых записей; ответ — массив объектов с полями time, datetime. Для авторизованного клиента: POST http://localhost:3001/api/service_bookings с телом { service_id, master_id, pet_id (опционально), scheduled_at (ISO), contact_method, notes }; заголовок Authorization: Bearer <JWT>; ожидается 201 и объект записи с user_id текущего пользователя. GET http://localhost:3001/api/service_bookings с тем же JWT должен возвращать только записи этого пользователя (благодаря ownerField и readOwn). Проверка прав: POST к /api/services без токена или с ролью client — 403; с токеном groomer или admin — 201 при валидном теле. Проверка ошибок: невалидный id в GET (например, /api/services/0) — 404; создание service_booking без обязательных полей (service_id, scheduled_at) или с временем вне слотов — 400 или 500 в зависимости от валидации.

**Пошаговый сценарий для модуля услуг.** Сначала GET /api/services — проверка, что возвращается список с полями name, category, type, price, duration, image. Затем GET /api/services/1 — детали первой услуги. GET /api/master_services?service_id=1 — список связей; по master_id из ответа можно запросить GET /api/masters/:id. Для теста записи на услугу: авторизация клиента (POST /api/auth/login), затем GET /api/service_bookings/slots?date=2026-02-20&service_id=1 для выбора времени, затем POST /api/service_bookings с телом { service_id: 1, master_id: 1, scheduled_at: "2026-02-20T10:00:00.000Z", contact_method: "по звонку" } и заголовком Authorization. В ответе — 201 и объект с id, user_id (текущий пользователь), service_id, status: "pending". Затем GET /api/service_bookings с тем же токеном — в списке должна быть созданная запись. Создание коллекции «MARS GROOM — Услуги» в Postman с папками «Чтение», «Слоты», «Запись», «Права» упрощает регрессионное тестирование при изменениях в API.

## 3.3. Общая часть — тестирование в Postman

Подход к тестированию в Postman общий для всех ресурсов, подключённых через createEntityRouter: для любого эндпоинта проверяются GET списка (с page, limit и фильтрами в query), GET по id, при необходимости POST/PUT/DELETE с учётом роли. Формат запросов и ответов единый: тело в JSON, ответы с полями success, data, при ошибке — success: false, error; для списков — pagination. Для авторизованных сценариев используется переменная token и заголовок Authorization: Bearer {{token}}; токен получается один раз через POST /api/auth/login и сохраняется в переменных окружения коллекции. Коллекции можно группировать по модулям («Курсы», «Услуги», «Пользователи» и т.д.), но сами запросы строятся по одному шаблону: метод, URL, опциональное тело, проверка статуса и тела ответа. Общая часть тестирования включает проверку health (GET /api/health), авторизации (POST /api/auth/login, сохранение токена), при необходимости CORS и обработку ошибок (404 «Запись не найдена», 403 «Нет прав», 400 при невалидных параметрах, 500 при сбое сервера). Валидация id (validateIdParam) на бэкенде возвращает 400 при нечисловом или неположительном id, что также проверяется в Postman. Таким образом, общая методология тестирования в Postman единообразна для модулей курсов и услуг и для всего API приложения.

# ПУНКТ 4. РАЗРАБОТКА УНИВЕРСАЛЬНОГО ХУКА

## 4.1. Модуль курсов — применение универсального хука

Универсальный хук useEntity обеспечивает единообразную работу фронтенда с любым CRUD-ресурсом API, в том числе с курсами и всеми связанными сущностями модуля курсов (расписание, преподаватели, отзывы, модули, запись на курс).

**Использование на страницах курсов.** На CourseDetailPage хук вызывается несколько раз: useEntity<CourseRow>('courses', { fetchListOnMount: false, id: courseId, fetchItemOnMount: !!courseId, enabled: !!courseId }) — загрузка одного курса по id из URL; useEntity<ScheduleRow>('course_schedules', { fetchListOnMount: false }) с последующим refetchList({ course_id: courseId, limit: 20 }) для расписания по курсу; useEntity<ReviewRow>('reviews', { fetchListOnMount: false }) и refetchList({ course_id: courseId, type: 'course', limit: 50 }) для отзывов; useEntity для course_instructors, masters, course_modules с параметрами listParams { course_id: courseId }. Таким образом, одна и та же абстракция используется для загрузки одной записи (курс) и списков с фильтрами (расписание, отзывы, преподаватели, модули), без дублирования логики запросов и обработки ошибок. На странице списка курсов применяется useEntity<CourseRow>('courses', { fetchListOnMount: true, listParams: { limit: 50 } }), что даёт list, loadingList, loadingListError и возможность отобразить карточки или индикатор загрузки и сообщение об ошибке. Для записи на курс форма может вызывать create из useEntity('course_bookings', …) с телом { course_id, course_schedule_id, notes }; хук сам отправит POST и при успехе обновит локальный список (добавит новую запись в начало), а состояния creating, createError позволяют показывать индикатор отправки и сообщения об ошибках.

**Преимущества для модуля курсов.** Типизация (CourseRow, ScheduleRow и т.д.) даёт подсказки и проверку полей в IDE; единый формат ответов API (success, data, pagination) обрабатывается внутри хука; состояния loadingItem, loadingItemError, creating, createError позволяют показывать скелетоны и сообщения без ручного управления флагами. За счёт этого фронтенд модуля курсов остаётся простым и консистентным при добавлении новых полей или сущностей (например, тесты курсов, прогресс): достаточно подключить ещё один useEntity с нужным именем ресурса и параметрами, без написания отдельных функций загрузки и обработки ответов.

## 4.2. Модуль услуг — применение универсального хука

В модуле услуг useEntity используется для загрузки списка услуг, одной услуги по id, списка мастеров, связей master_services и при необходимости списка записей пользователя и создания новой записи на услугу.

**Использование на страницах услуг.** ServicesListPage: useEntity<ServiceRow>('services', { fetchListOnMount: true, listParams: { limit: 100 } }); из хука берутся list (массив услуг), loadingList, loadingListError; список фильтруется на клиенте по категории и поисковой строке. ServiceDetailPage: useEntity для services с id и fetchItemOnMount для загрузки одной услуги и одновременно fetchListOnMount: true с listParams { limit: 100 } для списка (похожие услуги); useEntity('masters', { fetchListOnMount: true, listParams: { limit: 50 } }); useEntity('master_services', { fetchListOnMount: true, listParams: { service_id: serviceId } }). По listParams с service_id бэкенд возвращает только связи по данной услуге, что соответствует фильтрации в universalController (query-параметры передаются в where). MasterDetailPage: useEntity('masters', { id: masterId, fetchItemOnMount: true }), useEntity('groomer_portfolio', { listParams: { master_id: masterId } }) для загрузки мастера и его портфолио. На странице записи (ServiceBookingPage) для отправки заявки используется create из useEntity('service_bookings', …), что избавляет от ручного вызова api.post и обновления состояния; состояния creating, createError позволяют показывать индикатор отправки и сообщения об ошибках.

**Преимущества для модуля услуг.** Один и тот же хук обслуживает и публичный каталог (services), и детальную страницу (item + списки masters, master_services), и страницу мастера (masters, groomer_portfolio), и создание записей (service_bookings), сохраняя единообразие типов и обработки ошибок. При добавлении новых полей в таблицу services или новых сущностей достаточно обновить типы на фронте и при необходимости добавить новый роутер на бэкенде; вызов useEntity с новым именем ресурса даёт полный CRUD-интерфейс без дублирования кода запросов.

## 4.3. Общая часть — универсальный хук useEntity

Универсальный хук useEntity реализован один раз в проекте (hooks/useEntity.ts) и принимает имя сущности (строку, соответствующую пути API: courses, services, course_schedules, service_bookings, masters и т.д.) и опции (fetchListOnMount, listParams, id, fetchItemOnMount, enabled). Он возвращает list, item, pagination, refetchList, refetchItem, create, update, remove и все флаги загрузки и ошибок (loadingList, loadingItem, loadingListError, loadingItemError, creating, createError, updating, updateError, deleting, deleteError). Внутри хук использует единый api-клиент и формирует запросы GET/POST/PUT/DELETE к базовому пути /api/{entityName}; для списка параметры listParams и переданные в refetchList(params) объединяются в URLSearchParams и подставляются в GET-запрос. Ответ разбирается на data и pagination (для списка) или data (для одной записи) и записывается в состояние; при ошибке выставляются соответствующие Error и пустые данные. В create и update при успешном ответе локальный list и item обновляются (добавление новой записи в начало списка или замена по id), что обеспечивает актуальный UI без дополнительного refetchList. В useEffect при монтировании при включённых fetchListOnMount и fetchItemOnMount вызываются refetchList() и refetchItem(id). Модули курсов и услуг не реализуют собственных хуков для работы с API — они только вызывают useEntity с разными именами ресурсов и типами (CourseRow, ServiceRow и т.д.). Общая часть включает типы ApiListResponse, ApiItemResponse, ApiErrorResponse (в api/types) и соглашение о формате ответов бэкенда (success, data, pagination), что обеспечивает совместимость фронтенда и бэкенда для любых текущих и будущих сущностей приложения. Таким образом, общая реализация хука обеспечивает и модулю курсов, и модулю услуг единый способ загрузки данных, мутаций и отображения состояний загрузки и ошибок без дублирования кода на каждой странице.
